## 1. 시스템 구조 (Architecture)

```
[브라우저 / Vanilla JS]
   │  Google OAuth(GIS 토큰) → YouTube Data API v3 직접 호출
   │   - subscriptions.list (구독 목록)
   │   - videos.list?myRating=like (좋아요)
   │   - commentThreads / playlistItems (댓글·영상 제목)
   │  (선택) Google Takeout 업로드: comments.csv / watch-history.json
   │
   ├── 활동 신호 집계(채널별 댓글·좋아요·시청·마지막 시청일)
   ▼
[Flask 서버 / app.py]
   POST /api/analyze
     LRU 캐시 조회 → 원시 점수 → Recency Decay → Min-Max 정규화
       → Binary Search 그룹 분류 → Heap Sort 정렬
       → 카테고리 분류 → 코사인 유사도 → Union-Find 커뮤니티 / Kruskal MST
   POST /api/recommend, /api/persona, /api/cleanup → Upstage Solar LLM 프록시
   ▼
[브라우저] 결과 렌더링 + 필터 + force-directed 취향 지도(SVG)
```

- **프론트엔드**가 YouTube API를 직접 호출하고(토큰은 브라우저 보관), **백엔드**는 점수·그래프 연산과 LLM 프록시를 담당하는 **Frontend + Server** 2계층 구조.
- 점수·정렬·그래프 등 **알고리즘 핵심 연산은 서버(`app.py`)** 에, 시각화·발굴 로직은 **프론트(`static/app.js`)** 에 배치.

## 3. LLM(Upstage Solar) 활용

서버가 Solar API(`UPSTAGE_API_URL`, 모델 `solar-pro2`)를 프록시로 호출하며, 키는 `.env`의 `UPSTAGE_API_KEY`로만 주입한다(코드 하드코딩 없음). 용도:

1. **채널 추천** — 점수·카테고리 분포와 video-first 발굴 후보를 근거로 추천 사유 생성.
2. **페르소나 분석** (`/api/persona`) — 점수·카테고리 분포를 시청 취향 페르소나 리포트로 해석.
3. **구독 정리** — 휴면/저관심 채널에 대한 정리 제안.

LLM은 **점수·정렬·그래프 계산에는 관여하지 않는다.** 수치 분석은 결정론적 알고리즘이 수행하고, LLM은 그 결과를 자연어로 해석·추천하는 역할로 분리했다(재현성·검증성 확보).

### 4-2. 정렬 — Heap Sort (강의 03) + 안정 정렬

- **위치**: `app.py:110-144` (`_rank_key`, `_heapify`, `heap_sort_desc`)
- **구현**: 배열 기반 **이진 최대 힙**을 직접 구성(`_heapify`)해 점수 내림차순 정렬. 비교 키 `_rank_key`는 `(점수, 구독개월, -입력순서)` 튜플 — 점수가 같으면 구독 기간, 그것도 같으면 **원래 입력 순서**로 깨므로 **안정 정렬(stable)** 이 보장된다.
- **복잡도**: 힙 구성 O(n) + 추출 O(n log n) = **O(n log n)**, 제자리 정렬 O(1) 추가 공간.
- **강의 연계**: 비교 기반 정렬의 하한 Ω(n log n), 힙 정렬의 in-place 특성, 정렬 안정성 개념을 그대로 적용.

```python
def _rank_key(ch):
    return (ch["score"], ch.get("subMonths", 0), -ch.get("_order", 0))
# 입력 순서 인덱스 _order로 동점도 결정론적으로 정렬 → 안정성 확보
```

### 4-3. 탐색 — Binary Search (강의 04, 분할 정복)

- **위치**: `app.py:159-167` (`get_group`) + 경계값 `GROUP_BOUNDARIES = [20, 50, 80]`
- **구현**: 정규화된 0~100 점수가 어느 구간에 속하는지 **이진 탐색**으로 판정해 4그룹(🌱/👋/😊/🔥)으로 분류. `lo/hi`를 좁히는 표준 lower-bound 형태.
- **복잡도**: 경계 k개에 대해 **O(log k)**.
- **강의 연계**: 정렬된 경계 배열에 대한 분할 정복식 구간 탐색.

### 4-4. 해시 테이블 (강의 10) — LRU 캐시·해시맵·집합

- **LRU 캐시**: `app.py:47-66`. `OrderedDict`(해시맵+삽입순서 연결)로 **get/set O(1)**, 용량 초과 시 `popitem(last=False)`로 최소 사용 항목 축출. 캐시 키는 채널 ID에 활동·최근성 신호를 결합 → 입력이 바뀌면 자동 무효화. 강의의 *해시 충돌·로드팩터·O(1) 평균* 개념과 LRU 캐시 운용을 적용.
- **해시맵 집계**: 채널별 좋아요/댓글/시청 횟수를 `dict`/`Map` 키로 누적(평균 O(1)).
- **집합 기반 중복 제거**: 추천 결과에서 이미 구독한 채널·중복 ID·중복 이름을 `Set`으로 거른다(`static/app.js`의 `dedupeRecommendations`). 해시 집합의 평균 O(1) 조회 활용.
- **카테고리 결과 캐시**: 분류 결과를 버전 키로 캐싱해 재계산 방지.

### 4-5. 그래프 — Union-Find / 연결요소 (강의 11·12)

- **위치**: `app.py:312-333` (`UnionFind`), 적용 `app.py:384-410`
- **구현**: **서로소 집합(Disjoint Set)** 을 **경로 압축(path halving)** + **랭크 기반 합치기**로 구현. 코사인 유사도 ≥ 0.35인 채널쌍을 간선으로 보고 union → **연결요소(connected components)** 가 곧 "취향 커뮤니티". 각 커뮤니티는 최빈 카테고리를 대표 라벨로, 최고 점수 채널을 대표 채널로 표기.
- **복잡도**: 연산당 **거의 O(α(n)) ≈ O(1)** (역 애커만), 간선 처리 O(E·α(n)).
- **강의 연계**: 서로소 집합 자료구조, 경로 압축·union by rank 최적화, 그래프 연결요소.

```python
def find(self, x):
    while self.parent[x] != x:
        self.parent[x] = self.parent[self.parent[x]]  # 경로 압축
        x = self.parent[x]
    return x
```

### 4-6. 그래프 — Kruskal 최대 신장 포레스트 / MST (강의 05 그리디 + 11·12)

- **위치**: `app.py:412-422`
- **구현**: 유사도 간선을 **가중치 내림차순으로 정렬**한 뒤, Union-Find로 사이클을 만들지 않는 강한 간선부터 채택 → 커뮤니티별 **최대 신장 포레스트(MST)**. 결과 간선이 "취향 지도"의 연결선이 된다. (최소가 아니라 **최대** 신장 트리 = 유사도가 큰 간선 우선이라는 점만 다름.)
- **복잡도**: 간선 정렬이 지배적, **O(E log E)**.
- **강의 연계**: Kruskal MST의 **그리디 선택 성질**(가장 좋은 간선 우선, 사이클 배제)을 그대로 적용. 그리디(강의 05)와 그래프(강의 11·12)가 결합된 대표 사례.

```python
for s, i, j in sorted(edges, key=lambda e: -e[0]):  # 유사도 내림차순(그리디)
    if uf2.union(i, j):                              # 사이클이면 버림
        mst_edges.append({...})
```

### 4-7. 그래프 시각화 — Force-directed 레이아웃 (강의 11·12 응용)

- **위치**: `static/app.js`의 `drawTasteMap` (force-directed 시뮬레이션)
- **구현**: 노드 간 반발력(∝1/d²) + MST 간선의 스프링 인력 + 중심 인력을 약 200회 반복, 속도 감쇠(0.82)로 수렴시켜 SVG에 취향 지도를 배치. 그래프를 화면에 표현하는 응용.
- **복잡도**: **O(반복수 · n²)**.

### 4-9. 미적용·근사 항목 (정직한 한계 표기)

- **동적 계획법(강의 06)**: 고전적 DP(LCS·배낭 등)는 본 프로젝트에 직접 쓰지 않았다. 다만 LRU 점수 캐시는 **메모이제이션**(중복 계산 회피)과 개념적으로 같은 아이디어다.
- **탐색 트리(BST/RBT, 강의 09)**: 자기균형 트리는 직접 구현하지 않았고, 대신 **힙**(Heap Sort)과 **해시 테이블**(LRU)을 사용했다.
- **NP-완전성·근사(강의 13·14)**: 채널 군집화는 일반적으로 NP-hard에 가까운 문제이나, 본 프로젝트는 **유사도 임계값 + 그리디(MST/Union-Find)** 로 다항 시간에 **근사**한다. 엄밀한 근사비 보장은 범위 밖.

## 5. 그 외 핵심 구현 (보조 기법)

- **Recency Decay**(`app.py:87-95`): 마지막 시청 후 경과일 d에 대해 `score·e^(-0.005d)` 지수 감쇠 — 최근 관심 가중.
- **카테고리 분류**(키워드 규칙 + 정규식): 한글은 부분 문자열, 영문/숫자는 **단어 경계 정규식**으로 오탐 방지, core(가중 3)/general(가중 1) 합산 후 정규화 벡터화.
- **video-first 발굴**(`static/app.js`): 관심 키워드로 영상을 검색해 **channelId 빈도**를 집계, 2회 이상 등장 채널을 우선해 후보 풀 구성(단발 오탐 억제).
- **지수 백오프 재시도**: YouTube API 429 응답 시 `2^i초` 대기 재시도.
- **Takeout 파서**: watch-history.json/comments.csv를 정규식·스트림 파싱으로 채널 신호 추출(추가 API 비용 0).

## 6. 한계 및 향후 개선 (Limitations)

- **API 할당량**: YouTube Data API 일일 쿼터 제약. Takeout 하이브리드로 일부 완화하나 대규모 구독에는 한계.
- **카테고리 분류**: 키워드 규칙 기반이라 신조어·다국어 채널에서 오분류 가능 → 임베딩 기반 분류로 확장 여지.
- **유사도 그래프 O(n²)**: 구독 수가 매우 많으면 비용 증가 → 근사 최근접 이웃(ANN)으로 개선 가능.
- **테스트 부재**: 점수·그래프 알고리즘 단위 테스트 미비 → 회귀 검증 자동화 필요.
- **LLM 의존**: 추천/페르소나 품질이 Solar 응답에 의존 → 프롬프트·캐싱 고도화 여지.
