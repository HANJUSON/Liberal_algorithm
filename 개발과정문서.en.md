## 1. Project Overview

A single-page web app that, after connecting to your own YouTube account via Google OAuth, synthesizes your subscribed channels by **comments · likes · subscription duration · watch history** into an **interest score (0–100)** and automatically classifies them into 4 groups (🔥/😊/👋/🌱).
On top of that, it is a taste-analysis tool combining **Upstage Solar LLM**-based channel recommendation, persona analysis, and subscription cleanup, plus **category classification · cosine-similarity recommendation · taste communities (Union-Find) · taste map (MST)**.

- **Frontend**: Vanilla JS + HTML/CSS (single page), force-directed SVG graph visualization
- **Backend**: Python Flask (`/api/analyze`, LLM proxy, graph analysis, etc.)
- **LLM**: Upstage Solar
- **Auth**: Google Identity Services (GIS) token client, YouTube Data API v3

## 3. Weekly Development Details

### Week 09 — Planning & Setup
- After topic brainstorming, settled on "YouTube subscription channel interest analysis."
- Created a public GitHub repo, wrote the initial README, applied for Solar/NIM APIs.

### Week 10 — Requirements & Screen Design
- Defined 3 core features: ① interest scoring & auto-grouping, ② Solar-based channel recommendation, ③ subscription cleanup & persona analysis.
- Investigated data sources: decided on a hybrid strategy of YouTube Data API v3 + Google Takeout (comments.csv, watch-history.json).
- Designed a draft scoring method (weighted sum of comments · likes · subscription duration).

### Week 11 — Prototype
- Completed a working prototype in a single `youtube-analyzer.html`: OAuth login → subscription-list call → screen rendering.
- Validated the API call flow and UI layout.

### Week 12 — MVP (Backend + Algorithms)
- Separated the prototype into a **Flask server (`app.py`) + static frontend (`static/app.js`, `style.css`)** structure.
- **Implemented core algorithms**:
  - Activity summation + **dormant-channel 0-point short-circuit**
  - **Min-Max normalization** → scaling scores to 0–100
  - **Binary Search**-based threshold 4-group (🔥/😊/👋/🌱) classification
  - **Heap Sort** descending ordering
  - Input-dependent key **LRU cache** to cut re-analysis cost
- Reflected **Recency Decay** (weighting recent watches) when Takeout (watch history) is uploaded.
- **Started persona analysis & subscription cleanup features** (commit merged into 6/2 `58cae7a`).

## 4. Issue Tracking

| # | Issue | Cause | Resolution |
|---|------|------|------|
| 1 | Comment-count API accuracy limit | YouTube API does not provide exact per-channel comment counts | Exact value when `comments.csv` is uploaded, API approximation otherwise |
| 2 | Cost/latency on repeated analysis | Re-calling with identical inputs | Introduced an input-dependent key **LRU cache** |
| 3 | Dormant channels distort scores | Channels with 0 activity skew normalization | Separated out via **0-point short-circuit** |
| 4 | Insufficient recommendation accuracy | Limits of channel-level recommendation | Fully overhauled into **video-first discovery** |
| 5 | Unstable recommendation ordering | Ambiguous tie-handling / sort criteria | Category classification + **cosine similarity**, stable-sort secondary key |
| 6 | Recommendations skewed to one category | Similarity alone lacks diversity | **Category-diversity correction** + watch-history-based recommendation combined |
| 7 | Risk of taste-graph work conflicting with mainline | Graph feature touches `app.py` · UI broadly | Developed on a **separate `feat/taste-graph` branch**, then merged |
| 8 | Personal data exposed in the repo | Test watch/search history committed | Tidied `.gitignore` + untracked |

## 5. Summary of Applied Algorithms (addressing requirement #5)

| Algorithm concept | Application location | Purpose |
|---------------|-----------|------|
| Sorting (Heap Sort + stable-sort secondary key) | Descending sort by interest score, tie stabilization | Result ranking |
| Search (Binary Search) | Threshold-based 4-group classification | O(log n) interval determination |
| Hashing / cache (LRU) | `/api/analyze` input-key cache | Cut re-analysis cost |
| Normalization (Min-Max) | Scaling scores to 0–100 | Comparability across channels |
| Weighted decay (Recency Decay, `exp(-λ·d)`) | Watch-history-based weighting | Reflect recent interest |
| String matching (category classification) | Keyword-rule-based channel categorization | Generate taste vectors |
| Vector similarity (cosine similarity) | Category-vector-based channel recommendation | Discover taste-similar channels |
| **Graph — Union-Find (path compression · rank)** | Clustering connected components of the similarity graph | Derive **taste communities** |
| **Graph — Kruskal Maximum Spanning Forest (MST)** | Connecting channels by strongest-similarity-first | Build the **taste map** tree |
| Visualization (force-directed layout) | SVG node layout · render | Visualize the taste map |

## 6. Retrospective

**What went well**
- Incremental development: prototype (Week 11) → MVP (Week 12) → LLM extension (Week 13) → taste-graph refinement (Week 14).
- Met all requirements (web + LLM, FE/BE separation, use of Solar, **multiple algorithms including graph algorithms**, English presentation).
- Practical problem-solving such as bypassing API limits with the Takeout hybrid.
- The later taste-graph work (Union-Find/MST) was **isolated on the `feat/taste-graph` branch** then merged, reducing mainline conflicts.

**What was lacking / directions for improvement**
- Commits concentrated late, making it hard to gauge work distribution → commit in smaller units more often, use branches constantly from earlier on.
- Branch isolation was applied to only some features → need to standardize per-feature branches + PR review for all features.
- Lack of automated tests → regression verification relied on manual effort.
