## 1. System Architecture

```
[Browser / Vanilla JS]
   │  Google OAuth (GIS token) → calls YouTube Data API v3 directly
   │   - subscriptions.list (subscription list)
   │   - videos.list?myRating=like (likes)
   │   - commentThreads / playlistItems (comments · video titles)
   │  (optional) Google Takeout upload: comments.csv / watch-history.json
   │
   ├── activity-signal aggregation (per-channel comments · likes · watches · last watch date)
   ▼
[Flask server / app.py]
   POST /api/analyze
     LRU cache lookup → raw score → Recency Decay → Min-Max normalization
       → Binary Search group classification → Heap Sort ordering
       → category classification → cosine similarity → Union-Find communities / Kruskal MST
   POST /api/recommend, /api/persona, /api/cleanup → Upstage Solar LLM proxy
   ▼
[Browser] result rendering + filters + force-directed taste map (SVG)
```

- A **two-tier Frontend + Server** design where the **frontend** calls the YouTube API directly (the token stays in the browser) and the **backend** handles score/graph computation and LLM proxying.
- The **core algorithmic computations — scoring, sorting, graphs — live on the server (`app.py`)**, while visualization and discovery logic live on the **frontend (`static/app.js`)**.

## 3. LLM (Upstage Solar) Usage

The server calls the Solar API (`UPSTAGE_API_URL`, model `solar-pro2`) as a proxy, with the key injected solely via `UPSTAGE_API_KEY` in `.env` (no hard-coding in the code). Uses:

1. **Channel recommendation** — generates recommendation rationale based on score/category distribution and video-first discovery candidates.
2. **Persona analysis** (`/api/persona`) — interprets the score/category distribution as a viewing-taste persona report.
3. **Subscription cleanup** — suggests cleanup for dormant / low-interest channels.

The LLM **does not participate in score/sort/graph computation.** Numeric analysis is performed by deterministic algorithms, and the LLM is separated out to interpret and recommend on top of those results in natural language (ensuring reproducibility and verifiability).

### 4-2. Sorting — Heap Sort (Lecture 03) + Stable Sort

- **Location**: `app.py:110-144` (`_rank_key`, `_heapify`, `heap_sort_desc`)
- **Implementation**: An array-based **binary max-heap** is built directly (`_heapify`) to sort scores in descending order. The comparison key `_rank_key` is a tuple `(score, subscription months, -input order)` — ties are broken by subscription duration, and if those are equal, by **original input order**, guaranteeing a **stable sort**.
- **Complexity**: Heap construction O(n) + extraction O(n log n) = **O(n log n)**, in-place with O(1) extra space.
- **Lecture link**: Directly applies the Ω(n log n) lower bound of comparison-based sorting, the in-place nature of heap sort, and the concept of sort stability.

```python
def _rank_key(ch):
    return (ch["score"], ch.get("subMonths", 0), -ch.get("_order", 0))
# Tie-break deterministically by the input-order index _order → ensures stability
```

### 4-3. Search — Binary Search (Lecture 04, Divide and Conquer)

- **Location**: `app.py:159-167` (`get_group`) + boundaries `GROUP_BOUNDARIES = [20, 50, 80]`
- **Implementation**: Uses **binary search** to determine which interval a normalized 0–100 score falls into, classifying into 4 groups (🌱/👋/😊/🔥). A standard lower-bound form narrowing `lo/hi`.
- **Complexity**: **O(log k)** for k boundaries.
- **Lecture link**: Divide-and-conquer interval search over a sorted boundary array.

### 4-4. Hash Tables (Lecture 10) — LRU Cache · Hashmap · Set

- **LRU cache**: `app.py:47-66`. Using `OrderedDict` (hashmap + insertion-order linkage) for **O(1) get/set**, evicting the least-recently-used item with `popitem(last=False)` when capacity is exceeded. The cache key combines the channel ID with activity/recency signals → automatically invalidated when the input changes. Applies the lecture's concepts of *hash collisions, load factor, average O(1)* together with LRU cache operation.
- **Hashmap aggregation**: Accumulates per-channel likes/comments/watch counts via `dict`/`Map` keys (average O(1)).
- **Set-based deduplication**: Already-subscribed channels, duplicate IDs, and duplicate names are filtered out of recommendation results with a `Set` (`dedupeRecommendations` in `static/app.js`). Leverages the average O(1) lookup of hash sets.
- **Category result cache**: Caches classification results by a version key to avoid recomputation.

### 4-5. Graph — Union-Find / Connected Components (Lectures 11·12)

- **Location**: `app.py:312-333` (`UnionFind`), applied at `app.py:384-410`
- **Implementation**: A **disjoint set** implemented with **path compression (path halving)** + **union by rank**. Channel pairs with cosine similarity ≥ 0.35 are treated as edges and unioned → the **connected components** become "taste communities." Each community is labeled by its most frequent category and represented by its highest-scoring channel.
- **Complexity**: **near O(α(n)) ≈ O(1)** per operation (inverse Ackermann), O(E·α(n)) for edge processing.
- **Lecture link**: Disjoint-set data structure, path-compression / union-by-rank optimizations, and graph connected components.

```python
def find(self, x):
    while self.parent[x] != x:
        self.parent[x] = self.parent[self.parent[x]]  # path compression
        x = self.parent[x]
    return x
```

### 4-6. Graph — Kruskal Maximum Spanning Forest / MST (Lecture 05 Greedy + 11·12)

- **Location**: `app.py:412-422`
- **Implementation**: Similarity edges are **sorted by descending weight**, then Union-Find adopts the strongest edges that do not create a cycle → a **maximum spanning forest (MST)** per community. The resulting edges become the connectors of the "taste map." (The only difference from a minimum spanning tree is that this is a **maximum** spanning tree = higher-similarity edges are prioritized.)
- **Complexity**: Dominated by edge sorting, **O(E log E)**.
- **Lecture link**: Directly applies the **greedy choice property** of Kruskal MST (best edge first, exclude cycles). A representative case combining greedy (Lecture 05) and graphs (Lectures 11·12).

```python
for s, i, j in sorted(edges, key=lambda e: -e[0]):  # descending similarity (greedy)
    if uf2.union(i, j):                              # discard if it forms a cycle
        mst_edges.append({...})
```

### 4-7. Graph Visualization — Force-directed Layout (Application of Lectures 11·12)

- **Location**: `drawTasteMap` in `static/app.js` (force-directed simulation)
- **Implementation**: Repulsion between nodes (∝1/d²) + spring attraction along MST edges + center attraction are iterated ~200 times, converged with velocity damping (0.82) to lay out the taste map in SVG. An application of representing a graph on screen.
- **Complexity**: **O(iterations · n²)**.

### 4-9. Not-Applied / Approximate Items (Honest Limitations)

- **Dynamic programming (Lecture 06)**: Classical DP (LCS, knapsack, etc.) was not used directly in this project. That said, the LRU score cache is conceptually the same idea as **memoization** (avoiding redundant computation).
- **Search trees (BST/RBT, Lecture 09)**: Self-balancing trees were not implemented directly; instead a **heap** (Heap Sort) and a **hash table** (LRU) were used.
- **NP-completeness · approximation (Lectures 13·14)**: Channel clustering is generally close to an NP-hard problem, but this project **approximates** it in polynomial time via a **similarity threshold + greedy (MST/Union-Find)**. A rigorous approximation-ratio guarantee is out of scope.

## 5. Other Core Implementations (Auxiliary Techniques)

- **Recency Decay** (`app.py:87-95`): Exponential decay `score·e^(-0.005d)` over days elapsed `d` since last watch — weighting recent interest.
- **Category classification** (keyword rules + regex): Korean uses substring matching, while English/numbers use **word-boundary regex** to prevent false positives; core (weight 3) / general (weight 1) are summed and then normalized into a vector.
- **Video-first discovery** (`static/app.js`): Searches videos by interest keywords and aggregates **channelId frequency**, prioritizing channels appearing 2+ times to form a candidate pool (suppressing one-off false positives).
- **Exponential backoff retry**: On a YouTube API 429 response, retries after waiting `2^i seconds`.
- **Takeout parser**: Extracts channel signals from watch-history.json / comments.csv via regex / stream parsing (zero additional API cost).

## 6. Limitations and Future Improvements

- **API quota**: The YouTube Data API daily quota is a constraint. The Takeout hybrid mitigates it somewhat, but large subscription counts hit limits.
- **Category classification**: Being keyword-rule based, it can misclassify neologisms and multilingual channels → room to extend to embedding-based classification.
- **O(n²) similarity graph**: Cost grows with very large subscription counts → could be improved with approximate nearest neighbor (ANN).
- **Lack of tests**: Unit tests for the score/graph algorithms are insufficient → regression-verification automation is needed.
- **LLM dependence**: Recommendation/persona quality depends on Solar responses → room for prompt/caching refinement.
